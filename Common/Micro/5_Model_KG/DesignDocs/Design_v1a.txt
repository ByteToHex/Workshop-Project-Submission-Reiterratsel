**B - Mamdani Rule Induction**
- B0: Use frozen annual DuckDB data as the Mamdani base input.
- B1: Store Mamdani rule definitions and membership boundaries in Neo4j.
- B1a: Use repo-root `.env` for Neo4j connection settings.
- B1b: Seed one local model graph from `mamdani_rule_seed.json`.
- B2: Implemented Mamdani inputs are:
- B2a: `ICR`
- B2b: `GEARING`
- B2c: `DSCR`
- B2d: `REFI_RISK`
- B2e: `PAYOUT_RATIO`
- B2f: `FFO_COVERAGE`
- B2g: `NET_DEBT_EBITDA`
- B2h: `NULL_COUNT`
- B3: `NULL_COUNT` is a derived meta-risk input, not a raw financial ratio.
- B3a: It is derived from annual `calc_status = 'MISSING_INPUT'`.
- B3b: Implemented fuzzy memberships are:
- B3b1: `healthy` trapezoid `[0, 0, 1, 3]`
- B3b2: `watch` triangle `[2, 5, 8]`
- B3b3: `distress` trapezoid `[6, 9, 19, 19]`
- B3c: Implemented null-related rules are:
- B3c1: `R10`: `NULL_COUNT = distress` -> output `high`, weight `0.65`
- B3c2: `R11`: `NULL_COUNT = watch` AND `ICR = watch` -> output `high`, weight `0.65`
- B3d: `non_ok_count` is currently persisted for diagnostics and app display only. It is not currently a Mamdani input or rule antecedent.
- B4: Mamdani preprocessing must use both `metric_value` and `calc_status`.
- B4a: Handle `NEGATIVE_BASE` as a distress-state override for applicable metrics.
- B4b: Handle `DISTRESS_BASE` as a distress-state override for applicable metrics.
- B4c: Handle `LOW_DENOMINATOR`, `PARTIAL`, and `CLIPPED_SOURCE_SHARE` as lower-confidence inputs through confidence scaling where applicable.
- B4d: Keep preprocessing outputs separate from raw warehouse values.
- B5: Run fuzzy inference in Python.
- B5a: Read rules from Neo4j after seeding from `mamdani_rule_seed.json`.
- B5b: Fuzzify annual inputs.
- B5c: Apply rule firing and aggregation.
- B5d: Defuzzify to final Mamdani score with the implemented confidence-aware blend toward neutral when activation confidence is weak.
- B6: Output `distress_score_mamdani` in range `0-1` per ticker-period.
- B6a: Freeze each Mamdani score until the next annual checkpoint.
- B6b: Persist results to downstream table `reit_fuzzy.fact_fuzzy_cache`.
- B6c: Persistence is mandatory for auditability and app/runtime reuse.
- B6d: Write Mamdani persisted outputs to separate parquet shard `fuzzycache.parquet`.
- B6e: Keep persisted outputs separate from `fact_metric_value`, `metrics.parquet`, and raw financial shards.
