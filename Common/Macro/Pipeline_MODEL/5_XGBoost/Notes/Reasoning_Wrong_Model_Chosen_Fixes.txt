## The core problem, precisely stated

The current `choose_winner` uses **F1 as the primary gate with no tolerance**. Any difference in F1, even 0.001, decides the winner before AUC or RMSE are even looked at. This is wrong for two reasons:

**1. F1 is threshold-dependent. AUC is not.**

F1 is calculated by drawing a hard line at zero: positive prediction = "up", negative = "down". That threshold is arbitrary. A model with slightly higher F1 might just be predicting "up" more aggressively — which is exactly what happened with Optuna on 10d (positive rate 0.52 vs DEAP's 0.42). Its higher F1 came from more aggressive positive calling, not genuine superiority.

AUC evaluates the model's ability to *rank* up-days above down-days across all possible thresholds. It's threshold-independent and therefore a more honest measure of true directional skill.

**2. No tolerance means noise controls the outcome.**

F1 differences of 0.01–0.03 between two stochastic optimisers on a 150-row holdout are within the noise band of the single test window. Treating them as decisive is overfitting to one fold's quirks.

---

## The fix — one function only, nothing else changes

Replace the current `choose_winner` with this:

```python
# Tolerance for "effectively tied" — differences smaller than this are treated as equal.
# Set relative to typical metric noise on ~150-row holdouts.
_AUC_TOLERANCE = 0.005   # ~0.5pp AUC difference is noise on this sample size
_RMSE_TOLERANCE = 0.002  # ~0.2pp RMSE difference is noise

def choose_winner(
    optuna_result: HoldoutResult,
    deap_result: HoldoutResult,
    signed_target: bool,
) -> str:
    o = optuna_result.summary
    d = deap_result.summary

    if signed_target:
        # Primary: AUC — threshold-independent directional skill
        auc_diff = o["auc"] - d["auc"]
        if auc_diff > _AUC_TOLERANCE:
            return "optuna"
        if auc_diff < -_AUC_TOLERANCE:
            return "deap"

        # Secondary: RMSE — magnitude quality, penalises poor calibration
        rmse_diff = o["rmse"] - d["rmse"]
        if rmse_diff > _RMSE_TOLERANCE:
            return "deap"   # DEAP has lower (better) RMSE
        if rmse_diff < -_RMSE_TOLERANCE:
            return "optuna"

        # Tiebreak: F1 — directional balance at zero threshold
        return "optuna" if o["f1"] >= d["f1"] else "deap"

    # Unsigned targets: unchanged logic
    if o["rmse"] < d["rmse"] - _RMSE_TOLERANCE:
        return "optuna"
    if d["rmse"] < o["rmse"] - _RMSE_TOLERANCE:
        return "deap"
    return "optuna" if o["r2"] >= d["r2"] else "deap"
```

## Why this specific order (AUC → RMSE → F1) for your use case

| Priority | Metric | Why |
|---|---|---|
| 1st | AUC | Threshold-independent, directly measures directional ranking quality — what you said matters most |
| 2nd | RMSE | Magnitude quality — your stated secondary priority, also catches collapsed predictions |
| 3rd | F1 | Directional balance at zero threshold — useful tiebreak but biased by positive rate |

## Why the tolerances matter, especially for future horizons

When you move to 3d, 7d, 15d, or the abnormal returns model, the test set size and signal-to-noise ratio will both change. Without tolerances, a 0.002 AUC difference on a 60-row test window will decide the winner — that's pure luck. The `_AUC_TOLERANCE = 0.005` means "only switch winner if one model is meaningfully better, not just marginally different." On a larger abnormal returns dataset you could tighten this to 0.002 or even 0.001 since more rows make small differences more reliable.

**(1) How different is tolerance from fuzzy logic?**

They're related in spirit but different in mechanism.

Tolerance is a hard binary with a dead zone. Below the threshold, models are declared equal and you move on. Above it, one wins completely. It's still a crisp decision — just with a buffer zone added. Think of it as "ignore differences smaller than X, otherwise decide normally."

Fuzzy logic is different in that it never makes a fully crisp decision at any point. Instead of asking "is A better than B?", it asks "how much better is A than B?" and produces a *degree* of membership — a number between 0 and 1. Each metric contributes partially to the final score depending on how large the difference is, and all metrics are combined simultaneously rather than evaluated in a strict priority sequence.

The key structural difference: tolerance still has a priority ranking (AUC first, then RMSE, then F1). Fuzzy logic has no ranking — all metrics contribute simultaneously, weighted by how decisive their differences are.

---

**(2) What fuzzy encoding would look like here**

Instead of "DEAP wins on AUC therefore done", you'd compute a fuzzy superiority score for each metric, then aggregate:

```python
import math

def fuzzy_superiority(val_a: float, val_b: float, scale: float) -> float:
    """
    Returns a score in (0, 1) indicating how much a is better than b.
    0.5 = tied. >0.5 = a is better. <0.5 = b is better.
    scale controls how quickly differences become decisive.
    """
    diff = val_a - val_b
    return 1.0 / (1.0 + math.exp(-diff / scale))

def choose_winner(
    optuna_result: HoldoutResult,
    deap_result: HoldoutResult,
    signed_target: bool,
) -> str:
    o = optuna_result.summary
    d = deap_result.summary

    if signed_target:
        # Each metric: score > 0.5 means optuna is better, < 0.5 means deap is better
        # scale = typical noise magnitude for that metric on ~150-row holdout
        auc_score  = fuzzy_superiority(o["auc"],  d["auc"],  scale=0.015)  # decisive at ~0.03 diff
        rmse_score = fuzzy_superiority(d["rmse"], o["rmse"], scale=0.005)  # note: lower rmse is better, so reversed
        f1_score_v = fuzzy_superiority(o["f1"],   d["f1"],   scale=0.020)

        # Weighted combination: direction first, magnitude second, f1 tiebreak
        weights = {"auc": 0.55, "rmse": 0.30, "f1": 0.15}
        combined = (
            weights["auc"]  * auc_score +
            weights["rmse"] * rmse_score +
            weights["f1"]   * f1_score_v
        )
        return "optuna" if combined >= 0.5 else "deap"

    # Unsigned: rmse primary, r2 tiebreak
    rmse_score = fuzzy_superiority(d["rmse"], o["rmse"], scale=0.005)
    r2_score   = fuzzy_superiority(o["r2"],   d["r2"],   scale=0.020)
    combined   = 0.70 * rmse_score + 0.30 * r2_score
    return "optuna" if combined >= 0.5 else "deap"
```

The `scale` parameter on the sigmoid controls how fast a metric becomes decisive. A difference equal to `scale` gives ~73% confidence to one side. A difference of `3x scale` gives ~95% confidence — essentially decisive. So setting `scale=0.015` for AUC means a 0.015 AUC gap gives moderate preference, a 0.045 gap is nearly certain.

Verification on your 10d case:
```

RMSE diff = 0.0095, scale=0.005  →  fuzzy score ~0.98 in DEAP's favour
F1   diff = 0.0268, scale=0.020  →  fuzzy score ~0.26 in DEAP's favour (Optuna better)

Combined = 0.55*(0.30) + 0.30*(0.98) + 0.15*(0.26)
         = 0.165 + 0.294 + 0.039 = 0.498  →  DEAP wins (just barely)
```

**(3) Which is preferred here?**

**Tolerance is the right choice for this problem.** Here's why:

Fuzzy logic introduces two new things you have to justify: the weights (0.55, 0.30, 0.15) and the scales (0.015, 0.005, 0.020). These are not derivable from your data — they're judgment calls. With a single holdout window and a noisy macro target, you cannot empirically validate whether AUC should be weighted 0.55 or 0.60. You'd be tuning a meta-model on one data point, which is exactly the kind of overfitting you're trying to avoid.

The fuzzy approach also obscures what actually decided the outcome. When DEAP wins by 0.002 in the combined fuzzy score, you cannot tell which metric drove it without decomposing the calculation. With tolerance, the logic is transparent: "AUC difference was 0.013, exceeded tolerance of 0.005, DEAP wins, done."

Fuzzy encoding becomes genuinely useful when you have enough holdout windows to calibrate the scales and weights empirically — which is exactly the walk-forward scenario you noted for the abnormal returns model. At that point you'd have 10–20 test windows, could fit the scale parameters to observed metric noise, and the weighted combination would be more defensible. For now, with one fold and a noisy signal, tolerance gives you the same directional correction with far less unjustified complexity.
